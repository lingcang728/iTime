# iTime 仓库工作指引

## 完成后的发布闭环（强制）

每次完成本仓库的一轮修改任务后，除非用户当前明确要求不要提交、不要推送或不要打包，否则必须在同一轮任务中依次完成：

1. 对最终源码运行与改动风险相称的验证；发布前至少执行项目的完整验证门禁。
2. 检查完整 diff，只暂存本轮范围内的更改，并按可独立验证的功能切片提交。不得从含未提交源码的工作树打包。
3. 将当前分支推送到其已配置的远端上游，并确认远端已包含本轮最终 commit。
4. 从已推送且工作树干净的 commit 重新执行 `npm run tauri:build`，生成 Windows 安装包。
5. 校验 NSIS 安装包存在且为本轮新产物，记录其完整路径、修改时间、文件大小与 SHA-256；再次确认工作树干净。
6. 只有提交、推送、重新打包和产物校验全部成功后，才能向用户报告任务完成。

如果打包过程意外产生受 Git 跟踪的变化，必须先审查并提交、推送这些变化，再从新的远端 commit 重新打包。用户当前消息、系统或开发者指令与本规则冲突时，以更新且更高优先级的指令为准。
