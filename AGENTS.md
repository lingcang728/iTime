# iTime 仓库工作指引

## 完成后的发布闭环（强制）

每次完成本仓库的一轮修改任务后，除非用户当前明确要求不要提交、不要推送或不要打包，否则必须在同一轮任务中依次完成：

1. 对最终源码运行与改动风险相称的验证；发布前至少执行项目的完整验证门禁。
2. 执行 `npm run package:release`。该命令必须强制重新构建，并从同一次构建同步覆盖 `release/iTime.exe` 与 `release/iTime_<version>_x64-setup.exe`；不得只更新其中一个，也不得复用旧产物。
3. 校验 `release` 中这两个文件都存在、修改时间属于本轮构建、与 Cargo 构建源文件 SHA-256 一致，且目录中没有第三个 EXE 发布文件。
4. 检查完整 diff，只暂存本轮范围内的源码、配置和两个 `release` 发布文件，并按可独立验证的功能切片提交。两个发布文件必须进入同一轮最终提交，不能被 `.gitignore` 排除。
5. 将当前分支推送到其已配置的远端上游；确认远端最终 commit 与本地一致，并确认远端树同时包含 `release/iTime.exe` 和对应版本的 Setup。
6. 再次确认工作树干净。只有验证、两个文件同步更新、提交、推送和远端校验全部成功后，才能向用户报告任务完成。

`release/iTime.exe` 是可直接运行版本，`release/iTime_<version>_x64-setup.exe` 是安装包；二者都是必须版本化并推送的正式发布产物。用户当前消息、系统或开发者指令与本规则冲突时，以更新且更高优先级的指令为准。
